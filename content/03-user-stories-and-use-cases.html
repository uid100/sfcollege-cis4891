Describing the system

[Top-Level] User Requirements

Identifying, decomposing and managing formal requirements is an important part of software engineering.

System requirements are often supported by User Stories, Use-Cases, and Use-Case Diagrams. Don't confuse these. They sound similar but it is important to be able to explain how they are similar and to distinguish between them.

 

User Stories

The process of developing system requirements usually begins with identifying the user need for a solution or developing a problem description.

This is usually expressed as user stories.

The user story is a simple statement expressing a user or customer's need or desire for a solution. It should be expressed from the user's perspective

The user story follows a standard 3-part format, indicating:

  1. role
  2. request
  3. result

 

For example,

  • As a potential customer
  • I need to see customer reviews
  • So that I can make an informed product selection

 

User stories are usually written in collaboration with a customer or end user and express operational needs from the user's point of view.

These are simple expressions and during the initial phase of capturing originating requirements as many of these should be gathered as possible to describe the goals of the system.

This standard format is easily recognizable and easy to follow.

  • As a [Persona]
  • I want/need [Functionality]
  • So that [Desired Outcome]

 

Notice that sometimes the solution the user is requesting may not be the best solution for the desired outcome. Also, understanding the user's role and intended operational environment is essential to developing a usable and useful solution.

Formally, User Stories should include acceptance criteria specifying starting condition, action, and demonstrated objective. This is not always included at this stage.

Use Cases

In simple terms, a use case is an account of how a person who uses a certain process will achieve a goal. In technical terms, it is the description of the interaction between system and actors. The product of this process is a document containing all the steps followed by a user to achieve a goal.

(from: https://www.digitalnatives.hu/blog/user-story-vs-use-case/)

A Use-Case is a textual description of how a person will use a system to accomplish an objective.

There are many template examples for Use Cases. Although these may differ, and may be tailored to meet a particular need or audience, these will generally include:

  • Brief description (or title)
  • The [relevant] system
  • The actor(s) including those initiating or responding to events
  • Scenario usually will include the pre-existing conditions and (all) relevant action sequences
  • Expected result

 

Use cases can be quite detailed and should describe all of the actions and measurable results required to meet the user requirements.

The Use-Case may derive from one or more User Stories.

A Use-Case is not the same thing as a User Story.

A Graphical Summary

The Unified Modeling Language (UML) is a formal standard (currently v2.5) specification which defines 14 types of engineering diagrams, 7 behavioral, and 7 structural

One of these formal diagram definitions is the use case diagram which is a graphical representation of a set of use-cases depicting their interaction with actors either inside or outside of the system boundary.

Notice

  • The system boundary indicating which activities take place inside and outside of the system. The system boundary is represented by a box around the use case(s)
  • The actors, in this case all of them are indicated outside of the system boundary. The actors are a generalized persona. If multiple instances of a role are expected, only one actor is indicated to represent that role. Actors do not need to be users, they may represent external systems or APIs. They are represented by a stick figure
  • Straight lines are drawn indicating interaction between actor and use case.
  • Relevant use cases are represented in the diagram. All use cases associated with the system do not need to be included in the diagram to make the diagram useful.
  • Use cases are represented with a labeled oval shape.
  • The use case diagram is not an activity diagram or flow chart. Activity is not indicated by "flow" lines between use cases.

 

The use-case diagram is not the same as a use-case
