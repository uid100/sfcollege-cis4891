Project Assignment

Preliminary Design Documents

This week, migrate your project description from the last module into a GitHub repository and create and include these preliminary design document artifacts

  • (10 pts) Description and README
  • (10 pts) Solution architecture diagram
  • (20 pts) Wireframe diagram(s)
  • (20 pts) User Stories
  • (20 pts) Initial Use Cases
  • (20 pts) Use Case Diagram

 

The design documents in your readme will tell a visitor a story and communicate your vision for your project application.

 

GitHub

Create a public repository in GitHub. The name should be helpful in describing your application (not "Student_capstone_project" or "Project_1")

Add a brief (1-line) description for your application. You can edit this using the gear icon next to About on the front page of the repository.

Add some meaningful topical keywords. These can be relevant to the solution domain (e-commerce, healthcare, fitness, ...) or to the technology you are using (mvc, javascript, api, sql-server, ...)

Add a Readme (README.md) file. This file uses Markdown format and becomes the front page for your project as a place to collect and reference your design docs.

Markdown Resources

  • GitHub Markdown
  • Markdown Cheatsheet (adam-p)

 

README.md

Your README file should be organized and contain logical headings such as Overview, User Stories, Use Cases

Copy and paste your project description from the last module, format as Markdown and edit as needed. It may be appropriate to add the document to a documents folder within the repository and only include select pieces of it, sufficient to describe your project to visitors.

Deconstruct your application into multiple smaller components. Consider: database, data access API, presentation layer, admin application, others? Create a simple architecture (block) diagram indicating the different components and interfaces (links) between them. Keep this simple, not a lot of detail required for this.

Save your component drawing as an image file and add it to your README.md. Search for "GitHub markdown cheatsheet" for hints to adding images (and headings and table of contents) to your README file.

Wireframe Sketches

Create wireframe concept drawings of your application to communicate an at-a-glance idea of your vision for the solution.

Your pages should have notes indicating logical navigation sequence and logical flow of interface elements. Indicate whether there are any default form values, default actions, and element ordering (will the user tab through the items?)

These diagrams are not expected to be polished or finished drawings. They should provide a logical view of the user's typical workflow from one screen to the next. Ideas and logic are important, artwork is not.

Use the drawing tool of your choice. My preference for this is the sketchy template in the free tool: Pencil Project from Evolus.

Draw.io is another good (free) choice.

Export your drawings as Jpeg images and save them to an images folder in your repository and link them to your README.

You may create as many sketches here as make sense to tell the story. One may be enough.

User Stories

Create (at least 5) User Stories to describe how the application is intended to be used.

The use-cases should be formatted correctly and consistently.

Add these with a section heading to your README.

Use-Cases

Write (at least 5) Use-Cases and add them to your README. These may be in separate documents with links in the README, or written directly in the README.md file.

There is flexibility on formatting here. There are many example templates online. You can use either a tabular format or paragraphs with subheadings.

The Use-Cases should describe the primary usage for the application.

(UML) Use Case Diagram

Add a use-case diagram to your README. The diagram should indicate system boundary, actors, use-cases, links between actors and use-cases. Use the correct symbols for the UML specification.

The correct use-cases should be included in the diagram (This might not inlude all of them)

Be sure to save as jpeg or other common supported image format. Add a heading to the README.

Submission

Add the URL link to your project repository (not to the README.md). The README docment displays by default when the repository is visited.

The README should contain

  • Concept description
  • Wire-frames (labeled as DRAFT)
  • Requirements
    • User Stories
    • Use-Cases
    • Use-Case Diagram

 

Create your GitHub repository to contain your design documents and the new content you will be creating. Use the content pages that I created in this module to get you started. These are not exhaustive. In addition to these, combine with other online resources and your own experience to create these design artifacts.

Be sure to check the rubric for the requirements before you submit your work.

In the text entry, enter the link to your project README. The content there should be easy to find and review. In the next module, you'll be adding to it and peer reviewing one another's work.
